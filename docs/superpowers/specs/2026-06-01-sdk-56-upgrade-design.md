# Design — Expo SDK 54 → 56 Upgrade (combined: parity + plugins + Router + Expo UI)

**Date:** 2026-06-01
**Status:** Approved (pending written-spec review)
**App:** `mobile/` (Vaultic — Expo, Expo Router, expo-sqlite, fully offline)
**Branch:** `sdk-56-upgrade`, based on `feat/asset-improvements` (stacks on the unmerged coin work)

## Goal

Upgrade the app from **Expo SDK 54** (RN 0.81 / React 19.1) to **Expo SDK 56** (RN 0.85 / React 19.2, released 2026-05-21), and — in the same branch — adopt three SDK 56 capabilities: **type-safe config plugins**, **new Expo Router features**, and **Expo UI components**. Work is sequenced migration-first: the app must boot at parity on SDK 56 before any feature adoption begins.

This is explicitly a **single combined project** (user decision), but internally phased so the foundation lands and is verified before features are layered on.

## Guiding principle — idiomatic, no workarounds

Use React Native and Expo APIs **as intended**. Prefer the official/new API and built-in configuration over custom shims, monkey-patches, or legacy bridges. Concretely:

- **Use the new expo-file-system `File`/`Directory` API**, not `expo-file-system/legacy`. `/legacy` is permitted only for a specific operation that genuinely has no new-API equivalent, and any such use must be called out and justified — never as a shortcut to avoid migrating.
- **Replace custom config plugins with built-in Expo configuration wherever SDK 56 supports it.** Each of the 3 plugins is re-evaluated against native Expo config (see Phase 2); a plugin is kept only if no first-class config option exists.
- Prefer Expo Router's intended primitives over re-implementing navigation behavior by hand.
- Prefer Expo UI / official components over bespoke equivalents where they fit.
- No version pinning to dodge a migration, no patch-package hacks; if something truly needs a workaround, surface it explicitly with the reason rather than hiding it.

## Constraints (from `mobile/CLAUDE.md` and environment)

- **Both-platform rule:** do not bump `expo`/`react-native` without verifying on iOS + Android. Dev machine is Windows (no local iOS simulator), so: **Android verified locally each phase; iOS verified via EAS cloud builds at the end.**
- **CNG workflow:** `android/` and `ios/` are gitignored and fully regenerated by `npx expo prebuild --clean`. Native tweaks live in 3 config plugins under `plugins/`. No committed native code to reconcile.
- **Offline-first:** no backend deps; `fetch()` only; expo-sqlite is the only persistence layer. The price/FX endpoints are unchanged by this upgrade.
- New feature strings go through `t()` in all 16 languages; RTL (ar, fa) respected.
- EAS is configured (`eas.json`: development/preview/production; iOS simulator cloud image; store submit for both).

## Approach

**In-place, direct 54 → 56, CNG-regenerated, phased.** Bump `expo` to `^56`, run `npx expo install --fix` to align every native dependency, `npx expo prebuild --clean` to regenerate native projects, run `npx expo-doctor`, then fix the (small, contained) breaking JS surfaces. Verify Android + tests at each phase; iOS via EAS at the end.

Alternatives rejected:
- **Stepwise 54→55→56:** ≈2× effort; Expo's upgrade guidance is cumulative and our breaking surface is small, so the direct jump is acceptable. (We still read BOTH the SDK 55 and SDK 56 upgrade guides when planning, since a direct jump must absorb 55's breaking changes too.)
- **Fresh scaffold + port:** pointless under CNG — prebuild already yields a clean native baseline.

## Breaking-change surface (audited in this codebase)

| Area | Files affected | Notes |
|---|---|---|
| expo-file-system new API | `lib/hooks/useBackup.ts` (only) | SDK 54 `FileSystem.*` calls → SDK 56 `File`/`Directory` API, or `expo-file-system/legacy` as a bridge. Single file. |
| Status bar / navigation bar API consistency | `app/_layout.tsx` (only) | SDK 56 unifies these APIs; re-check against the safe-area work just shipped. |
| Expo Router (drops React Navigation) | `app/_layout.tsx`, `app/(tabs)/_layout.tsx`, `app/(tabs)/index.tsx`, `app/(onboarding)/_layout.tsx`, `app/(onboarding)/index.tsx` | ~5 files. Biggest behavioral risk. |
| Native deps majors | n/a (managed) | gesture-handler 2→3, async-storage 2→3, reanimated 4.1→4.4, screens, safe-area, svg, datetimepicker — pulled by `expo install --fix`. |
| Test harness | `__mocks__/expo-sqlite.ts`, jest config | jest-expo 54→56 likely pulls jest 30; mock/setup may shift. |
| Config plugins | `plugins/withAppDelegatePublicAccess.js`, `plugins/withLocalizedAppName.js`, `plugins/withPrivacyManifest.js` | Migrate to typed API (Phase 2). |

## Phases

Each phase ends green (`npm test` + `npx tsc --noEmit` + Android boot/manual smoke of touched flows) before the next begins.

### Phase 1 — Core SDK bump to parity (THE GATE)
1. Read the official Expo SDK **55** and **56** upgrade guides + changelogs (cumulative). Pull exact API changes (file-system, status/nav bar, router) into the implementation plan.
2. `expo@^56`; `npx expo install --fix` (aligns RN 0.85, React 19.2, all `expo-*`→56.x, reanimated 4.4, gesture-handler 3, screens, safe-area, svg, async-storage 3, datetimepicker, jest-expo 56).
3. Update devDeps that jest-expo 56 requires (jest 30, `@types/jest` 30 if needed); reconcile `typescript` only if forced.
4. `npx expo prebuild --clean --platform android`; `npx expo-doctor` → resolve issues.
5. Fix breaking JS:
   - `lib/hooks/useBackup.ts` — migrate to the **new** expo-file-system `File`/`Directory` API (per the guiding principle; `/legacy` only if a specific op has no new-API equivalent, with justification).
   - `app/_layout.tsx` — status/navigation-bar API changes; re-verify against existing safe-area handling.
   - gesture-handler 3 / reanimated 4.4 / async-storage 3 / RN 0.85 / React 19.2 runtime + type fixes.
6. Repair tests/mocks for jest-expo 56 (esp. `__mocks__/expo-sqlite.ts`).
7. **Exit criteria:** `npm test` green, `npx tsc --noEmit` clean (no NEW errors beyond known pre-existing mock/test ones), app boots and core flows work on an Android emulator/device.

### Phase 2 — Config plugins: prefer built-in config, then type-safe for the rest
Per the guiding principle, first try to **eliminate** each custom plugin by using native Expo configuration; only convert what genuinely must remain to the typed `ConfigPlugin` API.
- **`withAppDelegatePublicAccess`** — Xcode-26 AppDelegate-access workaround (commit `80d8c1e`). SDK 56's Expo Modules changes very likely obsolete it. **Delete** if confirmed unnecessary.
- **`withLocalizedAppName`** — replace with Expo's built-in localized-app-name support (the `expo.locales` config / per-locale `CFBundleDisplayName`) if it covers the need; **delete the plugin** if so.
- **`withPrivacyManifest`** — replace with Expo's first-class iOS Privacy Manifest support (`ios.privacyManifests` in app config) if available in SDK 56; **delete the plugin** if so.
- Any plugin that must remain → convert to a typed `ConfigPlugin` (`.ts`).
- `expo prebuild --clean`; Android still green. (Plugin changes are iOS-native — re-verified on the EAS build in Phase 5.)

### Phase 3 — Expo Router new features
- Adopt the post-React-Navigation Router APIs across the ~5 router files, keeping the tabs + onboarding behavior at parity first, then enabling the new capabilities that add value.
- Verify navigation flows on Android.

### Phase 4 — Expo UI adoption
- Adopt Expo UI native components on **specific, bounded targets** (chosen in the plan — likely the picker/bottom-sheet UI in the asset form and settings), incrementally and screen-by-screen, preserving function and improving native feel. Not a blanket UI rewrite (YAGNI).

### Phase 5 — iOS verification (EAS)
- `eas build --profile development --platform ios` (simulator) and a `preview` build; smoke-test core flows (onboarding, assets/coins, zakat, debts, settings, backup/restore, app-lock) on iOS.
- Fix iOS-specific issues; produce a both-platform `preview` build before any release.

## Verification strategy

- **Per phase:** `npm test` (jest-expo), `npx tsc --noEmit`, Android boot + manual smoke of the flows touched in that phase.
- **Regression signal:** existing Playwright web E2E suite where applicable.
- **End-to-end:** EAS iOS build + manual smoke (Phase 5).
- **No mock prices**; price/FX/crypto endpoints unchanged — confirm they still resolve after the bump.

## Risks & unknowns

- **Router without React Navigation** — highest behavioral risk; mitigated by parity-first then enhance, and by the small file count.
- **expo-file-system rewrite** — contained to one file; `/legacy` is the fallback.
- **jest-expo 56 / jest 30** — test setup and the expo-sqlite mock may need rework; tests are the primary safety net so this is front-loaded in Phase 1.
- **Expo UI scope** — open-ended; bounded to named screens in the plan.
- **Two-major jump** — must absorb SDK 55 breaking changes too; addressed by reading both upgrade guides during planning.
- **iOS-only breakage invisible until Phase 5** — accepted trade-off of the Android-first decision; the config-plugin changes (Phase 2) are the most iOS-sensitive and will be re-checked on the EAS build. Removing iOS-native plugins (localized name, privacy manifest, AppDelegate access) carries the risk that the built-in replacement behaves slightly differently — Phase 5 must explicitly confirm: app display name per locale, privacy manifest presence, and a clean iOS launch.

## Branch & integration

- Branch `sdk-56-upgrade` is based on `feat/asset-improvements` (coin work), so the SDK upgrade and the coin feature ship together and the coin feature is migrated as part of this work. Final integration target is `main`.
